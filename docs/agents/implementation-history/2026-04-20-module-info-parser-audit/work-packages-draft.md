## WP-1 — Adopt `BuildMessages` in `ModuleInfoParser`

**Description:** Replace the three `echo` calls in `ModuleInfoParser::parseFile()` with `BuildMessages` calls, and change the YAML parse error path from re-throwing the exception to returning `null` (non-fatal).

**Scope:**
- `src/classes/Application/Composer/ModulesOverview/ModuleInfoParser.php` (framework)

**Deliverables:**
- `ModuleInfoParser` uses `BuildMessages::addError()` and `BuildMessages::addWarning()` instead of `echo`/`sprintf`
- YAML parse errors return `null` instead of re-throwing the exception
- `ParseException` import retained (still used in `catch` clause)

**Acceptance Criteria:**
1. No `echo` calls remain in `ModuleInfoParser::parseFile()`
2. A YAML parse error produces a `BuildMessages::addError()` call with source label `'ModuleInfoParser'` and returns `null`
3. Missing `moduleMetaData` produces a `BuildMessages::addWarning()` call
4. Incomplete `moduleMetaData` produces a `BuildMessages::addWarning()` call
5. `ModulesOverviewGenerator` still functions correctly (it already handles `null` returns)

**Estimated Complexity:** Low

**Notes:** This is a prerequisite for WP-2 — the non-fatal error handling ensures `KeywordGlossaryGenerator` can safely iterate over all files without aborting on a single bad YAML file. Independent of WP-4 and WP-5.

## WP-6 — Create framework-level `ModuleJsonExportGenerator`

**Description:** Create the generic, subclassable `ModuleJsonExportGenerator` in the framework that encapsulates the application-agnostic module JSON export workflow.

**Scope:**
- New file: `src/classes/Application/Composer/ModulesOverview/ModuleJsonExportGenerator.php` (framework)

**Deliverables:**
- Framework-level `ModuleJsonExportGenerator` with:
  - Constructor accepting `FolderInfo $rootFolder`
  - `generate(string $outputPath, bool $includeAll = false): void` method
  - Protected hook method `resolveModuleSource(ModuleInfo $module): string`
  - Protected hook method `resolveModuleBrief(ModuleInfo $module, string $sourcePath): ?string`
  - Integration with `ModuleContextFileFinder`, `ModuleInfoParser`, `KeywordGlossaryBuilder`, `ReadmeOverviewParser`
  - `DecorateGlossaryEvent` firing for custom glossary sections
  - JSON output with `generatedAt`, `modules`, `glossary`, and `glossarySections` keys

**Acceptance Criteria:**
1. `ModuleJsonExportGenerator` exists at `src/classes/Application/Composer/ModulesOverview/ModuleJsonExportGenerator.php`
2. The class is NOT `final` (subclassable)
3. `resolveModuleSource()` default returns `$module->getComposerPackage()`
4. `resolveModuleBrief()` default looks for `README-Brief.md` in the source path
5. Uses `KeywordGlossaryBuilder` for glossary building (not inline logic)
6. Uses `ReadmeOverviewParser` for README overview extraction
7. Fires `DecorateGlossaryEvent` and includes `glossarySections` in JSON output
8. `composer dump-autoload` has been run in the framework

**Estimated Complexity:** High

**Notes:** Depends on WP-3 (`KeywordGlossaryBuilder`) and WP-4 (`ReadmeOverviewParser` in framework). This is the largest WP but cannot be split further — the class is a single cohesive workflow and partial creation would not be testable. The `glossarySections` integration is new functionality: the HCP Editor's dead `collectGlossarySections()` was never called, so this is the first time glossary sections will appear in JSON output.
