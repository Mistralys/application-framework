### Quick Start Workflow

1. **Read** [context-documentation.md](docs/agents/project-manifest/context-documentation.md) — understand how knowledge is structured.

5. **Fall back to source code** only when the documentation does not cover the specific detail needed.

### CTX-Generated Documentation (`.context/`)

The `.context/` folder is in the **project root** and is generated by the [CTX Generator](https://github.com/context-hub/generator) from `context.yaml` configuration files distributed throughout the codebase. Regenerate with `composer build` (which includes autoload dump, module docs, and CTX generation as its final step).

> **Note:** `.context/` is committed to VCS so that agent search tools can discover its contents.
> After running the build, commit the updated generated files.

## 2. Manifest Maintenance Rules

| Change Made | Documents to Update |
|---|---|
| New module added | `module-context.yaml` (add module context), run `composer build`, update module table above |
| Module restructured | Module's `module-context.yaml`, run `composer build` |
| New coding convention established | `docs/agents/project-manifest/constraints.md` |
| Testing infrastructure changed | `docs/agents/project-manifest/testing.md` |
| New dependency added/removed | `composer.json`, run `composer dump-autoload` |
| Directory structure changed | Run `composer build` to refresh `.context/framework-file-structure.md` |
| Documentation strategy changed | `docs/agents/project-manifest/context-documentation.md` |
| Database schema changed | `docs/sql/` |
| New class file added/renamed | Run `composer dump-autoload` (classmap autoloading) |

## 3. Efficiency Rules — Search Smart

This codebase has **1,545+ class files**. Wasteful scanning is expensive. Follow this lookup order:

1. **Finding files?** Check `.context/framework-file-structure.md` or the module's `file-structure.md` FIRST.
2. **Understanding a module?** Read the module's `.context/` `overview.md` and `architecture-core.md` FIRST.
3. **Unfamiliar domain term?** Check `docs/agents/project-manifest/module-glossary.md` to map the term to the relevant module(s) before searching the codebase.
4. **Coding patterns?** Check `docs/agents/project-manifest/constraints.md` and `docs/agents/coding-patterns.md` FIRST.
5. **Only then** read source files.

## 4. Failure Protocol & Decision Matrix

| Scenario | Action | Priority |
|---|---|---|
| Ambiguous requirement | Use the most restrictive interpretation | MUST |
| Manifest/code conflict | Trust the manifest, flag code for fix | MUST |
| Missing CTX documentation for a module | Read the module's README and source directly, flag the gap | MUST |
| `.context/` docs seem inconsistent with codebase | Run `composer build` to refresh all generated docs | SHOULD |
| Untested code path | Proceed with caution, add test recommendation | SHOULD |
| Legacy (non-namespaced) class encountered | Do not refactor unless explicitly asked; follow existing style in that file | MUST |
| Unsure if `array()` or `[]` syntax | Always use `array()` — this is a hard project rule | MUST |
| Need to add a new class file | Run `composer dump-autoload` after — classmap autoloading requires it | MUST |
| Found `tests/phpstan/` directory | This contains only bootstrap constants for `phpstan.neon` — use `composer analyze` to run PHPStan; never restore or create scripts there | MUST |

## 6. Testing Quick Reference

Full documentation: [testing.md](docs/agents/project-manifest/testing.md)

| Command | Purpose |
|---|---|
| `composer test-file -- <path>` | Run a single test file |
| `composer test-filter -- <pattern>` | Run tests matching a name pattern |
| `composer test-suite -- <name>` | Run a named test suite |
| `composer test-group -- <group>` | Run tests in a PHPUnit group |

### Choosing the right scope

1. **Changed a single class?** → `composer test-file` with its test file.
2. **Changed a module?** → `composer test-filter -- ModuleName`.
3. **Unsure which tests cover a change?** → `composer test-filter -- ClassName`.

## 7. Static Analysis Quick Reference

Run PHPStan via Composer scripts — **never invoke `vendor/bin/phpstan` directly**.

| Command | Purpose |
|---|---|
| `composer analyze` | Run PHPStan against the codebase |
| `composer analyze-save` | Run PHPStan and write results to `phpstan-result.txt` |
| `composer analyze-clear` | Clear the PHPStan result cache |

> **Warning:** `tests/phpstan/` contains only a bootstrap constants file (`constants.php`) consumed by `phpstan.neon`. It is **not** a PHPStan runner or CLI tooling directory. Never restore, create, or execute scripts from this folder — always use the Composer commands above.

## 9. Developer Tools Quick Reference

### Interactive Developer Menu

Launch the menu-driven interface for common developer tasks from the project
root:

```bash
# Unix / macOS
./menu.sh

# Windows
menu.cmd

# Direct (any platform)
php tools/menu.php
```

If `vendor/` is missing the menu runs `composer install` automatically before
showing any options.

| Option | Action |
|---|---|
| 1 | Setup local environment (`php tools/setup-local.php`) |
| 2 | Build (`composer build`) |
| 3 | Run tests — sub-prompts for an optional filter pattern |
| 4 | Clear caches (`composer clear-caches`) |
| 5 | Seed test database (`composer seed-tests`) |
| 6 | PHPStan analysis (`composer analyze`) |
| 0 | Exit |

### Local Environment Setup

Interactive one-command setup for the local development environment:

```bash
composer setup
```

Prompts for database and UI settings, generates `test-db-config.php` and
`test-ui-config.php` from their `.dist.php` templates, creates the database if
it does not exist, imports `tests/sql/testsuite.sql`, and runs
`composer seed-tests` on completion.

Re-running is safe — existing values are shown as defaults and pressing Enter
preserves them.

The script can also be invoked directly:

```bash
php tools/setup-local.php
```

> **Note:** The generated config files (`test-db-config.php`, `test-ui-config.php`)
> are listed in `.gitignore` and must never be committed.
